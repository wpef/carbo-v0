# Feature Specification: Schema Write

**Feature**: 022-schema-write
**Created**: 2026-03-25
**Updated**: 2026-04-01
**Status**: Draft
**Depends on**: 008-destination-field-retrieval, 012-field-mapping

## User Scenarios & Testing

### User Story 1 - Add a new field to the destination (Priority: P1)

As a consultant, I can add a new field to the destination directly from the field mapping view. On the destination field list, I click an "Add field" button. I choose whether to create a brand new field or copy from a source field. If I copy from a source field, the following information is pre-filled: name, type, picklist values (if applicable), and description. If I create from scratch, I fill in these fields manually. The description can be generated by an LLM.

**Why this priority**: During migration planning, destination schemas are rarely complete. Consultants need to create missing fields directly from the mapping workspace without switching to another tool.

**Independent Test**: A consultant opens the field mapping view for Contact-to-Contact. They click "Add field" on the destination list, select "Copy from source field", pick "CustomField__c" from the source. Name, type (text), and description are pre-filled. The consultant adjusts the name, clicks Create. The field appears in the destination field list.

**Acceptance Scenarios**:

1. **Given** the field mapping view, **When** the consultant clicks "Add field" on the destination column, **Then** a creation form opens with the choice: "New field" or "Copy from source field".
2. **Given** "Copy from source field" is selected, **When** the consultant picks a source field, **Then** the form pre-fills: name, type, picklist values (if picklist), and description.
3. **Given** "New field" is selected, **When** the form opens, **Then** all fields are empty and must be filled manually.
4. **Given** the creation form, **Then** the following fields are available: name (required), type (required), picklist values (required if type is picklist), and description (optional).
5. **Given** the description field, **When** the consultant clicks a "Generate description" button, **Then** the system calls an LLM to generate a description based on: a metaprompt about the company and its market, information about the object type in question, the field name and type, and a few sample values (if GDPR permits).
6. **Given** valid field details, **When** the consultant submits the form, **Then** the field is created in the destination system and appears in the destination field list.
7. **Given** a field name that already exists, **When** the consultant submits, **Then** the system displays an error before or after the API call.

### User Story 2 - Modify a destination field (Priority: P1)

As a consultant, I click on a destination field card to open a detail modal. From this modal, I can modify the field's name, type, picklist values (if applicable), description, and group (property group on HubSpot, field group on Salesforce). I save the changes and they are applied to the destination system.

**Why this priority**: Destination fields often need adjustments (updated picklist values, corrected types, better descriptions) as the migration plan evolves. This must be possible in-place.

**Independent Test**: A consultant clicks on destination field "industry" (picklist). The detail modal shows current values. The consultant adds a new picklist value "SaaS", changes the description, and clicks Save. The changes are applied to HubSpot.

**Acceptance Scenarios**:

1. **Given** the field mapping view, **When** the consultant clicks on a destination field card, **Then** a detail popup opens showing editable field properties.
2. **Given** the detail popup, **Then** the following properties are editable: name, type, picklist values (if the field is a picklist), description, and group (property group on HubSpot, field group on Salesforce).
3. **Given** the consultant modifies field properties, **When** they click Save, **Then** the changes are applied to the destination system and the local schema snapshot is updated.
4. **Given** the destination system rejects the modification (e.g., type change not allowed), **When** the save fails, **Then** a clear error message is displayed.

---

### User Story 3 - LLM-generated field descriptions (Priority: P2)

When creating or modifying a destination field, the consultant can generate a description using an LLM. The generation uses context including: a metaprompt about the company and its market, information about the object type, the field name and type, and a few sample values (if GDPR permits). The consultant reviews and edits the generated description before saving.

**Why this priority**: Writing field descriptions is tedious but important for documentation and client validation. LLM generation speeds this up significantly.

**Independent Test**: A consultant creates a new field "annual_revenue" (number) on the Company object. They click "Generate description". The LLM produces: "Annual revenue of the company in their reporting currency. Used for segmentation and reporting." The consultant edits it slightly and saves.

**Acceptance Scenarios**:

1. **Given** the field creation or modification form, **When** the consultant clicks "Generate description", **Then** the system generates a description using the LLM with appropriate context.
2. **Given** the LLM generates a description, **When** it is displayed, **Then** the consultant can edit it before saving.
3. **Given** no LLM API key is configured, **When** the consultant clicks "Generate description", **Then** a message explains that LLM generation is unavailable and the description must be written manually.
4. **Given** the LLM call fails (timeout, error), **When** the generation fails, **Then** a clear error is displayed and the consultant can write the description manually.

---

### User Story 4 - Create new objects in the destination (Priority: P3)

As a consultant, I can create a new custom object in the destination system (specifying name and primary property) when the destination schema lacks an appropriate object for the migration.

**Why this priority**: Less frequent than field creation, but necessary for migrations involving custom objects that don't exist in the destination.

**Independent Test**: A consultant creates a custom object "Projects" in HubSpot with a primary property "project_name" (text). The object is created in HubSpot and appears in the destination object list.

**Acceptance Scenarios**:

1. **Given** a destination connection that supports schema writes, **When** the consultant triggers object creation, **Then** a form requests: object name and primary property definition.
2. **Given** valid object details, **When** the consultant submits, **Then** the object is created in the destination and becomes available in the object list.
3. **Given** a destination that does not support schema writes, **When** the consultant looks for the option, **Then** the feature is not available with a message explaining why.

### Edge Cases

- The destination system has tier/plan limitations on custom fields or objects: the system surfaces the error clearly (e.g., "Custom property limit reached").
- The property name conflicts with a reserved word: the system displays a clear error from the destination.
- The destination system is temporarily unavailable during a write: the system reports the error and allows retry.
- The consultant creates a field with a type not supported by the destination: the system validates type compatibility before submitting.
- Network drops during a write operation: the system reports the failure. The consultant can check the destination to see if the operation partially completed.
- The property group specified does not exist: the system either creates the group or reports an error, depending on the connector's behavior.
- The LLM generates a description with sensitive or incorrect content: the consultant always reviews before saving (the description is never auto-saved).
- Sample values for LLM context contain PII: the system only includes sample values when explicitly allowed by GDPR configuration.

## Requirements

### Functional Requirements

- **FR-001**: The system MUST check the canWriteSchema capability flag before offering schema write features. If false, schema write MUST NOT be available.
- **FR-002**: The system MUST allow adding new fields to destination objects from the field mapping view, with: name (required), type (required), picklist values (required if picklist), description (optional), and group (optional).
- **FR-003**: The system MUST offer a "Copy from source field" option that pre-fills name, type, picklist values, and description from a selected source field.
- **FR-004**: The system MUST allow modifying existing destination field properties: name, type, picklist values, description, and group.
- **FR-005**: The system MUST offer LLM-generated descriptions using context: company/market metaprompt, object type, field name and type, and sample values (when GDPR-compliant).
- **FR-006**: If the LLM API key is not configured, the description generation button MUST be hidden or disabled with an explanatory message.
- **FR-007**: The system MUST allow creating new custom objects with: name and primary property definition.
- **FR-008**: The system MUST validate inputs before submitting: name uniqueness (against known schema), type compatibility (against destination's supported types), required fields.
- **FR-009**: The system MUST display clear error messages for all failure scenarios: name conflicts, tier limitations, invalid types, network errors, and system-specific errors.
- **FR-010**: Every schema write operation (success or failure) MUST be logged to the audit trail with: operation type, target, details, result, and timestamp (Constitution Principle VI).
- **FR-011**: After a successful schema write, the system MUST automatically refresh the local schema snapshot to include the newly created or modified elements.
- **FR-012**: The system MUST NOT allow schema writes on source connections, only on destination connections.

### Key Entities

- **SchemaWriteOperation**: Audit record of a schema modification. Fields: id, connectionId, operationType (CREATE_OBJECT | CREATE_FIELD | MODIFY_FIELD), targetObjectApiName, details (name, type, values, description, group), result (SUCCESS | ERROR), errorMessage (optional), createdAt.

## Success Criteria

### Measurable Outcomes

- **SC-001**: A field creation round-trip (submit form to confirmation) completes in under 10 seconds.
- **SC-002**: A field modification round-trip completes in under 10 seconds.
- **SC-003**: LLM description generation returns within 10 seconds.
- **SC-004**: "Copy from source field" pre-fills all available properties in under 1 second.
- **SC-005**: 100% of schema write operations are logged in the audit trail, including failures.
- **SC-006**: Pre-validation catches name conflicts and invalid types before making API calls.
- **SC-007**: Schema write features are invisible for connectors that do not support them.

## Assumptions

- Not all connector adapters support schema writes. The feature is gated by the canWriteSchema capability flag.
- Schema writes are only available for destination connections, not source connections.
- The connector adapter implements createObject, createField, and modifyField methods when canWriteSchema is true.
- Field types available for creation/modification are defined by the destination connector.
- The LLM for description generation uses the Claude API. The metaprompt about company/market context is configured at the project or plan level.
- GDPR compliance for sample values is a configuration setting. When disabled, the LLM generates descriptions without sample values.
- Field modification capabilities depend on the destination system's API (some systems may not allow type changes on existing fields).
