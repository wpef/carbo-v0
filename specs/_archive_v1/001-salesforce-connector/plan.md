# Implementation Plan: Salesforce Source Connector

**Branch**: `001-salesforce-connector` | **Date**: 2026-03-19 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/001-salesforce-connector/spec.md`

## Summary

Build a Salesforce source connector that authenticates via OAuth2, retrieves schema metadata
(objects, fields, types, constraints, FLS), displays paginated records with basic field stats,
and supports schema refresh with diff comparison. Uses jsforce as the Salesforce SDK, Next.js
Route Handlers for the API layer, and Prisma + SQLite for persistence.

## Technical Context

**Language/Version**: TypeScript 5.x (Next.js 14+ App Router)
**Primary Dependencies**: Next.js, jsforce v2.0+, Prisma ORM, Tailwind CSS, shadcn/ui
**Storage**: SQLite via Prisma (local-first for v0; migratable to PostgreSQL later)
**Testing**: Vitest (unit + integration), Playwright (E2E)
**Target Platform**: Web browser (local development, localhost)
**Project Type**: Web application (single Next.js project — frontend + backend unified)
**Performance Goals**: Schema browsing <2 min end-to-end; record preview first page <5s for 100k records
**Constraints**: Local-first (no cloud deployment for v0); read-only connector; single org per connection
**Scale/Scope**: Single consultant user; Salesforce orgs up to 500+ objects, 100k+ records per object

**Structure Decision**: Single Next.js project with unified frontend + backend. The `lib/connectors/salesforce/`
directory contains all business logic, isolated from the HTTP layer (API routes) and presentation layer
(components). This separation supports future Connector SDK extraction: the `lib/` layer is the candidate
for abstraction, while routes and components follow a repeatable pattern per connector.
